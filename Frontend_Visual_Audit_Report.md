# Frontend Visual & UI Integrity Audit

## 1. Executive Summary
The frontend has a usable product UI foundation: App Router layout, RTL-aware shell, Tailwind v4 semantic tokens, dark-mode variables, focus-trapped dialogs, skeleton utilities, and responsive table wrappers are present. The healthiest areas are the shared app shell and the newer tokenized dashboard/list components.

The largest remaining issue category is **UI Consistency & Theming (CON)**. Several older forms, analytics panels, nursery screens, and custom chart widgets still use literal `bg-white`, `text-slate-*`, hardcoded hex colors, inline color styles, and Recharts/SVG color arrays instead of the semantic tokens defined in `app/globals.css`. These defects are most visible in dark mode and make the design system harder to maintain.

There are also **Accessibility (A11Y)** and **Typography/Overflow (TYP)** risks in icon-only chat/activity controls, small touch targets, and dynamic text fields that do not consistently use truncation or `break-words`. Layout-breaking issues are mostly contained by `overflow-x-auto`, but several dense tables rely on 700-1120px minimum widths and therefore degrade into horizontal scrolling on phones.

This audit pass also fixed several shared UI defects before this report was written: sidebar/theme/pagination token usage, focus-visible states, CRM/asset table icon-button labels, hit areas, and dynamic-name truncation.

## 3. Global CSS & Configuration Risks
- **Tailwind Config / Global CSS:** Tailwind v4 is configured through `app/globals.css`, and semantic tokens are present. However, `globals.css` also overrides entire Tailwind neutral families and includes compatibility aliases for invalid legacy utilities (`--color-gray-255`, `--color-slate-655`, etc., lines 391-412). That keeps legacy classes rendering but hides debt and makes visual output harder to predict. The root neutral variables at lines 10-12 are also mis-indented, a minor maintainability signal.
- **Theme Support:** Theme support is partial. The core app shell and newer shared components use semantic variables, but many pages still hardcode light surfaces (`bg-white`, `bg-slate-50`), literal hex colors, and inline chart colors. Dark mode will work globally but remain visually inconsistent until older components are migrated to semantic tokens.

## 4. Priority Action Plan
1. Fix high-impact accessibility first: add `aria-label`, `aria-pressed`, `aria-expanded`, visible focus states, and 44px touch targets to chat controls, activity action buttons, image delete overlays, and tree expand/collapse controls.
2. Migrate legacy forms and dashboards to semantic tokens: start with `ContactForm`, `BreedingEntryForm`, `FlockAnalyticsDashboard`, nursery dashboard cards, and the older invoice/asset/team modals.
3. Reduce mobile table debt: add card layouts or column controls for the 900-1120px barn/watchlist tables, then add truncation/line clamps to dynamic labels across inventory, archive, nursery, and flock tables.
